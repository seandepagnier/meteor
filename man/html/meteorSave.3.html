NAME

meteorLoad, meteorSave  

SYNOPSIS

#include <stdio.h>
#include <meteor.h>

int meteorLoad(FILE *file, int fileformat);
int meteorSave(FILE *file, int fileformat);  

DESCRIPTION

The functions meteorLoad and meteorSave read and write respectively the current meteor data from a file stream. The fileformat parameter specifies the format to use, see FORMATS below for possible formats. If file is NULL then no operation is performed, only a test is made to verify the given fileformat is available.  

FORMATS

METEOR_FILE_FORMAT_TEXT
The first line has 3 integers. The first number is a mask of the point data to come. The masks are 1=Position, 2=Normals, 4=Colors, 8=Texcoords. For example, a value of 13 means Position, Colors and Texcoords, a value of 1 means only Position. The second and third numbers are the number of points and triangles. The next line is numeric data for first point in the meteor. The number of numbers on this line is based on the mask. Each point is separated by a newline. After all of the points the triangle data follows as space separated triples of point indexes. The last triangle is followed by two newlines. To see an example of this execute "meteor -nt8 -ex -c-".
METEOR_FILE_FORMAT_BINARY
This format is nearly identical to METEOR_FILE_FORMAT_TEXT except it ommits all whitespace and newlines, and instead of using ascii representations for numbers, 32bit integers, and 64bit ieee754 floats are used. This format is potentially less portable but has much faster load and save times and also stores more precision.
METEOR_FILE_FORMAT_WAVEFRONT
This format is currently only supported for writing and does not support colors. It might be used to import data into other applications.

 

RETURN VALUE

These functions return 0 on success and -1 on failure. meteorLoad may fail if the data read is not in the right format. Other errors include io errors, or use of an unsupported format. If a failure occurs, meteorError will be set.  

NOTES

These functions are convenience for reading and writing a meteor from disk, they are implemented entirely on top of meteorReadPoints, meteorReadTriangles, meteorWritePoints, and meteorWriteTriangles.
